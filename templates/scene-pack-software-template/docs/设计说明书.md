# 详细设计说明书

说明 Pack 的对象模型、流程设计、UI surface、适配声明和测试策略。

## 1. 业务概念架构设计

本 Scene Pack 采用严格解耦的 V3 六元运行架构设计，保障高并发场景下数据与状态绝对不交叉污染：

```text
  [ Scene Pack Spec (图纸) ]
             │
             ▼
  [ Business Object (人物/道具实例) ] <─── OCC state_version 乐观锁控制
             │
             ▼
  [ SceneFlowRun (流程运行实例) ] <─── run_id 彻底隔离
             │
             ▼
  [ TaskCandidate (任务/行动候选) ] <─── Task Governance 优先队列分区治理
             │
             ▼
  [ GatewayCandidate (执行动作候选) ] <─── Base Gate / OwnerDecision 主权确认
             │
             ▼
  [ Receipt Timeline (回执账本) ] <─── append-only 录像与签字，只追加不覆写
```

## 2. 数据结构设计

### 2.1 业务对象模型 (BusinessObject)

每个项目实例拥有唯一的 `object_id` 并强绑定一个不可变 `pack_version_ref`。所有的字段变更必须通过 `state_version` 进行版本控制，杜绝并发覆写冲突。

### 2.2 流程运行状态模型 (SceneFlowRun)

每个执行的流程生成全局唯一的 `run_id`，独立存储当前状态节点、运行事件 `run_events` 以及等待状态 `wait_state`。

## 3. 声明式 UI Surface 设计

描述 Pack dashboard、对象详情 surface、候选卡和事务胶囊如何映射到 Truzhen 标准 slot。前端视图只能映射到 standard UI slots 并读取 ReadModel 投影，一切修改意图必须作为 `GatedAction` 发送给后端，由 Base Core 门禁审核后触发。

## 4. 适配声明与外部集成

描述 adapter 的输入输出和候选类型。真实 provider 必须默认禁用，并由平台网关受控启用。实操中，真实动作必须提交到 `Execution Gateway`，由隔离沙盒 (Sandbox)、本地受控进程或人类接管 (Human Takeover) 执行，并输出不可篡改的 `Artifact` 和回执。

## 5. 并发与物理隔离设计

1. **immutable spec**：已启用的运行期流程强绑定特定 Pack 发布版本，升级 Pack 不会影响正在运行的实例。
2. **StepRun 隔离与幂等**：每个节点执行分配唯一租约 (Lease) 与 `idempotency_key`，超时后由引擎自动恢复。
3. **Namespace 物理隔离**：本 Pack 的 Adapter 及脚本只能在沙盒命名空间（如 `example_scene_pack_ns`）内运行与读取数据，绝对无法读取、改写其他 namespace 下的字段与事实。

## 2. R0：三标准与开跑硬门

1. 记录 OS/client/packs SHA、clean、Pack digest；跑 manifest/schema、JSON/Python、45 源文档/15 scope 全量分页、禁品、路径和 namespace 审计。
2. Manifest 明确 `SceneFlow completion != Transaction completion`，正式案件完成证明为执行、送达、整改复查 Receipt；lifecycle 仍为`已实现 -> 已接线`。
3. Runtime 回归必须证明：显式 KnowledgeMount 仍服从 query；无关辐射/案例不因 mount 混入固废查询；零相关结果为 `not_ready`；陈旧 candidate input 零增量；Owner 裁定可从 07/03 回放。
4. 运行 OS 定向/race/EGR、client 材料/历史定向与全量/typecheck/build/smoke；失败先修根因。

## 3. R1：唯一 C01 的完整候选生命周期

唯一项目 `C01-v18-固体废物露天堆放线索核查`，固定 transaction、Pack version/digest、浏览器 profile 和证据目录；不得新建项目逃避缺陷。

1. 直接空 registry 安装启用；核对 2 Role Pack、2 SlotBinding、15 mounts、45 源文档、version/digest 和 Receipt。
2. GUI 创建 C01。PDF 前 required 材料 missing/pending 时，角色请求 `role_review_evidence_not_ready/EvidenceGapCandidate`，模型/Candidate/Task/Receipt/外发/执行增量 0。
3. 上传合成 PDF，逐项补齐主体、辖区、物料性质、现场记录、监测检测、许可台账；上传不自动等于 verified，冲突与待人工核验继续阻断。
4. 材料齐备后各生成一次执法精英与挑剔律师候选。每个事实必须锚定 evidence；法律检索必须命中固废 query、案发 as-of、辖区和来源。注入无关辐射/水/案例 mount，模型上下文和引用中必须为 0；无相关法源时 `not_ready`，不得凭模型常识补法条。
5. 对一个材料/文书候选执行 Owner `needs_revision`，刷新、重启后从 06/07/03 恢复 PDF spine、候选、裁定和 Receipt；不得显示泛化“已确认”或丢失历史。
6. 形成处置与文书草稿候选，走最终 Owner/Base Gate，但不正式处罚、送达、删除、对外提交或执法执行。
7. SceneFlow end 只能显示“执法协助候选链收束”；05 Transaction 因执行/送达/整改复查正式 Receipt 为 0，必须保持未完成并列出缺口。

## 4. R2：Runtime 故障、停用与历史

1. 对缺 Provider、模型超时、断网、版本冲突、重复 PDF、旧请求和双 Tab逐项攻击；稳定 `blocked/not_ready/provider_missing`，零重复候选/模型/Receipt。
2. 停用后攻击 PDF、角色、文书和旧请求，03/06/07/08/09 业务增量 0；同一 C01 的材料矩阵、候选、needs_revision、Gate 和 Receipt 可只读回放。
3. 重新启用并重启 OS/client；只恢复新请求，停用期请求不延迟补写；项目固定原版本。
4. 390×844、1024×768、1440×900 全按钮、console/network、断网恢复留证；原始 refs 收在技术详情，人话状态准确。

# 徒真（truzhen）Pack 作者指南

你的用户是中国 3-50 人的中小企业团队。他们不关心你的技术栈，只关心三件事：这东西能帮我把哪件事办了、出了问题怎么办、值不值这个价。这份指南帮你把能力翻译成他们听得懂、敢购买的样子。

## 你适合做哪种作者

- **开源软件作者**：你维护 CLI、SDK、MCP Server、自动化脚本、数据处理或模型工具——它们可以变成企业随取随用的能力包。
- **行业高手**：你知道销售、客服、交付、法务、财税、本地服务里一件事该怎么推进——这些方法可以变成拿来就能开工的场景包。
- **角色设计者**：你能把服务风格、判断框架、沟通方法讲成可复用的东西——它们可以变成有你风格的角色包。

## 一个好 Pack 要说清六件事

1. **服务谁**：目标行业、团队规模、典型岗位。写得越具体，买的人越容易对号入座。
2. **解决什么**：用户不用懂底层软件，也能看懂这个 Pack 让哪件事变快、变稳、变得说得清。
3. **这个行业怎么说**：列出外行容易误解、团队内部默认不解释的关键术语、口径、关系和例外；写明来源、适用范围与谁核验。没有证据就标「待核验」，不要让 AI 猜。
4. **需要什么**：依赖哪些外部软件、账号、权限和知识来源；哪些内容需要人工核验。
5. **不能做什么**：哪些动作永远不会自动执行；外部能力没接通时，Pack 会诚实显示「未就绪」（技术口径为 `blocked / provider_missing / not_ready`），而不是假装能用。
6. **怎么留痕**：哪些关键动作会留下记录，让企业能回放核对发生了什么。

## 关于收益，说实话

作者可以通过徒真 Pack 市场获得平台分成。收益取决于 Pack 是否通过审核、是否经得起试装、是否真的解决目标客户的问题——我们不承诺固定收入，也请你不要向用户承诺。

经验之谈：先交付一个小而强的 Pack——目标客户清楚、边界清楚、跑得出真实记录——比一上来铺一个大而全的方案更容易通过审核，也更容易卖出去。

## 第一次投稿建议

第一次不要证明“我能做一个行业宇宙”，先证明“我能帮一类人办好一件事”。

- 开源作者可以先提交一个能力包，例如「把装修报价单 PDF 抽出项目、金额和风险项」。
- 行业高手可以先提交一个场景包小闭环，例如「家政咨询后，自动整理需求、生成方案草稿、提醒负责人确认」。
- 角色设计者可以先提交一个角色包，例如「客服质检主管，专门挑出话术里的承诺过度和遗漏信息」。

第一版不要求覆盖所有客户，也不要求接通所有 provider。它必须诚实说明适用范围、依赖条件、风险说明和负责人确认点。

## 最小可提交形态

- **能力包**：能力说明、外部依赖声明（ProviderRequirement）、风险说明、输入输出、确认门槛和留痕口径。
- **角色包**：角色用途、口吻、判断框架、知识引用、禁区和输出范围。
- **场景包**：manifest、流程、角色槽、角色包、能力引用、必要知识索引和 README。若场景包含歧义术语、计算 / 判断口径或高风险行业规则，还应提交相应 glossary / SOP / checklist / index 知识条目；暂缺时在 README 明示覆盖缺口——不阻塞最小起步，但不得宣称完整。

只有愿景、营销文案或 mock 演示，不能算可上架的 Pack。

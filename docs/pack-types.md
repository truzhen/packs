# 三类 Pack 怎么选

徒真（truzhen）只有三类 Pack：能力包、角色包、场景包。新增想法必须先归入其中一类；不能为了一个新点子发明第四种 Pack。

## 能力包

能力包描述“系统能调用什么能力”。适合开源软件作者、MCP Server 作者、CLI / SDK / sidecar 作者。

应声明：

- 能力名称、输入输出和使用限制。
- ProviderRequirement：需要哪些外部软件、账号、sidecar 或云服务。
- 风险等级、门控要求和失败状态。
- 调用后应该产生什么 Evidence / Receipt。

能力包不实现 provider，不保存凭据，不直接执行。

## 角色包

角色包描述“AI 应该以什么角色提出候选”。适合行业顾问、运营负责人、客服主管、交付经理等把判断框架沉淀成角色。

应声明：

- 角色职责、口吻、判断框架和质询方式。
- 允许引用的知识范围和候选输出范围。
- 禁止越权的动作：批准、发送、执行、正式记忆写入。
- 适合绑定到哪些 Role Slot。

角色包永远是 Proposer，不拥有审批权。

## 场景包

场景包描述“一个行业工作台怎么运转”。适合行业专家、垂直软件作者或服务团队。

应声明：

- 工作模式、事务流程、业务对象和状态。
- 角色槽、角色包、能力引用和 ProviderRequirement。
- Owner Gate、Base Gate、多角色对照、执行门、发送门。
- 证据、回执、知识来源和验收方式。

场景包不保存正式业务事实，不直接执行 provider；运行解释归徒真受控链路。

## 选择建议

| 你的资产 | 首选类型 |
| --- | --- |
| 一个 CLI、SDK、MCP Server、OCR、PDF、IM 或自动化工具 | 能力包 |
| 一个专家人设、服务风格、判断法、沟通法、质询法 | 角色包 |
| 一套行业流程、工作台、项目交付方法、客服闭环、合规流程 | 场景包 |

如果你不确定，先从最小能力包或角色包开始；等真实客户场景明确后，再组合成场景包。
